# Security

My SPACE is designed around a single threat model: the user's data should be unreadable to anyone who does not have the master password (Chrome) or the device (Android), including Google, the extension developer, and anyone with access to the synced Drive file. This page enumerates the security controls that realise that model.

## AES-GCM encryption

All secrets are encrypted at rest with AES-256-GCM, an authenticated encryption scheme that produces both ciphertext and a 128-bit authentication tag. The tag means any tampering with the ciphertext (bit-flipping, truncation) is detected at decrypt time and throws, rather than silently producing wrong plaintext.

- **Chrome** (`chrome-extension/src/offscreen/crypto.ts`): Web Crypto `crypto.subtle` does the encryption in the offscreen document. A fresh 12-byte random IV is generated per call via `crypto.getRandomValues`. See [cryptography](./systems/crypto.md).
- **Android** (`android/app/src/main/java/com/myspace/app/crypto/CryptoManager.kt`): `javax.crypto.Cipher` with `AES/GCM/NoPadding`, 128-bit tag length, IV generated by the Keystore provider.

Sync payloads (the full DB export) are also AES-GCM encrypted before upload to Google Drive, so Google only ever sees ciphertext.

## PBKDF2 key derivation

The Chrome extension derives the AES key from the user's master password using PBKDF2 with SHA-256 and **600,000 iterations**:

```ts
crypto.subtle.deriveKey(
  { name: 'PBKDF2', salt, iterations: 600_000, hash: 'SHA-256' },
  baseKey,
  { name: 'AES-GCM', length: 256 },
  false,              // not extractable
  ['encrypt', 'decrypt'],
)
```

600,000 iterations is well above the OWASP minimum (600k is the current recommendation as of 2023 for SHA-256 PBKDF2). A unique 16-byte salt per vault ensures that two users with the same password derive different keys, defeating rainbow tables and precomputation attacks. The derived `CryptoKey` is marked `extractable: false`, so even a compromise of the offscreen document's JavaScript context cannot read the raw key bytes.

The Android app does not use PBKDF2. Its key is generated inside the Android Keystore and bound to secure hardware, so there is no password to brute-force. See [cryptography](./systems/crypto.md) for the trade-offs.

## Vault auto-lock

The Chrome extension locks the vault (zeroes the in-memory key) after 15 minutes of inactivity by default. The timeout is configurable via `autoLockMs` in `chrome.storage.local` (set `0` to disable). Two independent timers enforce this:

1. **Offscreen timer** (`crypto.ts`): a `setTimeout` that fires `lockVault()`. Reset on every secret read/write via `resetLockTimer`.
2. **Side panel idle timer** (`App.tsx` `useIdleLock`): watches `mousemove`/`keydown` and sends `VAULT_LOCK` after the idle period.

Once locked, the key is gone and the user must re-enter their master password. There is no way to recover the key without the password; PBKDF2 is a one-way derivation.

The Android app has no auto-lock because its key lives in the Keystore, not in process memory. The device lock screen is the equivalent boundary.

## No servers, no analytics

There is no backend. The extension and the app talk only to:

- `www.googleapis.com` (Drive API and OAuth userinfo endpoint).
- `chromiumapp.org` (Chrome's internal OAuth redirect host).

No telemetry, analytics, or crash reporting SDKs are included. The `package.json` dependency list (`@electric-sql/pglite`, `lz-string`, `react`, `react-dom`) contains no tracking libraries. The Android manifest declares only `INTERNET`, `USE_BIOMETRIC`, and `USE_FINGERPRINT` permissions. `allowBackup` is `false` so the encrypted database is not included in Google cloud backups or device-to-device transfers.

This means there is no server that could be breached to leak user data, and no third-party SDK that could exfiltrate it.

## Drive appDataFolder isolation

Sync uses the `https://www.googleapis.com/auth/drive.appdata` scope, which grants access only to the application's hidden `appDataFolder`, not to the user's general Drive files. Other applications cannot see this folder, and the user cannot see it in the Drive web UI. The backup file is named `keyvault-backup.json` and is written via multipart upload with `parents: ['appDataFolder']` on first push.

The service worker (`chrome-extension/src/service-worker/index.ts`) caches the file ID in `chrome.storage.local` after the first push so subsequent pushes update the same file rather than creating duplicates.

## Cross-device password prompt

When pulling a backup created on a different device (or profile), the local salt will differ from the backup's salt. The pull handler detects this by comparing `backup.salt` to the local `vaultSalt` and, if they differ, returns `{ needsPassword: true }` instead of attempting to decrypt. The side panel then prompts for the master password and sends `SYNC_PULL_CONFIRM`, which re-derives a temporary key from the password and the **backup's** salt via `SYNC_DECRYPT_WITH_SALT`. The temporary key is used only for that one decrypt and is never stored. See [cryptography](./systems/crypto.md) for the full sequence.

This ensures the backup is decryptable on any device that knows the password, without ever transmitting the password or the derived key to Google.

## Markdown XSS sanitisation

Notes support markdown rendering via `chrome-extension/src/lib/renderMarkdown.ts`. Because the rendered HTML is injected via `innerHTML` in the note preview, the renderer must prevent stored XSS. It does so in three passes:

1. **Code blocks first** — fenced code blocks are extracted and their contents are HTML-escaped (`escapeHtml` replaces `&`, `<`, `>`) before any other processing.
2. **Script stripping** — `<script>...</script>` tags are removed with a regex.
3. **Event handler stripping** — all `on*` attributes (`onclick`, `onerror`, etc.) are stripped regardless of quoting style.
4. **Inline HTML removal** — the `inlineMarkdown` function strips any remaining `<...>` tags from inline text (`text.replace(/<[^>]+>/g, '')`), so only the markdown-emitted tags survive.
5. **Link sanitisation** — markdown links are only emitted if the href matches `^https?:\/\/`; otherwise the href is replaced with `#`.

This is a denylist-based approach layered on an allowlist (only the tags the renderer itself emits survive), which is reasonably robust for a single-user note app where the attacker and the victim are the same person. The content CSP (below) is the backstop.

## Content CSP policy

The manifest (`chrome-extension/manifest.json`) declares:

```json
"content_security_policy": {
  "extension_pages": "script-src 'self' 'wasm-unsafe-eval'; object-src 'self'"
}
```

- `script-src 'self'` — only scripts from the extension's own packaged bundle can execute. No inline scripts, no `eval`, no remote scripts.
- `'wasm-unsafe-eval'` — the sole relaxation, required for PGlite's WebAssembly module. This allows WASM compilation but not arbitrary `eval` of JavaScript strings.
- `object-src 'self'` — only the extension's own objects can be embedded (blocks `<embed>`, `<object>` from other origins).

Even if an XSS payload survived the markdown sanitiser, this CSP prevents it from loading an external script beacon or executing inline JS. The attacker could manipulate the DOM but could not exfiltrate data to a remote server via script injection.

## OAuth scope minimisation

The manifest requests three OAuth scopes:

| Scope | Purpose | Necessary? |
|-------|---------|-------------|
| `drive.appdata` | Read/write the hidden appDataFolder for sync | Yes, core sync feature |
| `userinfo.email` | Show the connected Google account email in Sync view | Cosmetic, but scoped to just email |
| `userinfo.profile` | Show the Google avatar in Sync view | Cosmetic, but scoped to profile fields only |

Notably absent: `drive.file` (would allow access to user-visible Drive files), `drive` (full Drive access), and any broader Google API scopes. The `host_permissions` are limited to `https://www.googleapis.com/*`, so the extension cannot fetch any other origin.

The Android manifest uses a custom-scheme redirect (`com.myspace.app://oauth2callback`) rather than embedding a WebView, so the OAuth consent and token exchange happen in the user's browser where they can verify the URL and scopes. See [Android app](./applications/android-app.md).

## Threat model summary

| Threat | Mitigation |
|--------|------------|
| Drive compromise / Google reads backup | AES-GCM ciphertext, key never sent to Google |
| Device theft (Chrome) | Vault auto-lock, key not persisted to disk |
| Device theft (Android) | Keystore hardware-bound key, `allowBackup=false` |
| Brute-force master password | PBKDF2 600k iterations + per-vault salt |
| XSS via note markdown | HTML stripping + event-handler stripping + CSP |
| Malicious extension injection | CSP `script-src 'self'` blocks remote scripts |
| Overly broad OAuth scope | `drive.appdata` only, plus cosmetic userinfo |
| Unintended cloud backup of DB (Android) | `allowBackup="false"` |
| Cross-device sync with wrong password | `SYNC_DECRYPT_WITH_SALT` never caches the temp key |
